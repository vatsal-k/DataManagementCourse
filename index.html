Data Ingestion — Getting Data In
▼
BatchStreamingCDC KafkaAPI PullWebhooks

Data ingestion is the entry point of any data pipeline. It determines latency, cost, and downstream freshness. The two fundamental modes are batch and streaming.

Batch vs Streaming

⚡ Batch vs Streaming Ingestion
DimensionBatch IngestionStreaming Ingestion
LatencyMinutes to hoursMilliseconds to seconds
ToolsAirflow, Cron, SparkKafka, Kinesis, Flink
CostLowerHigher (infra always on)
Use caseNightly reports, ETLFraud detection, alerts
ComplexityLowerHigher (ordering, exactly-once)

Change Data Capture (CDC)

CDC is a pattern to capture only the rows that changed in a source database, instead of full table scans. Tools like Debezium read the database transaction log (binlog) and emit events to Kafka.

Common Ingestion Sources

  • RDBMS: PostgreSQL, MySQL via JDBC or CDC
  • REST APIs: Polling or webhook push
  • Message queues: Kafka, RabbitMQ, SQS
  • Files: CSV, Parquet, JSON dropped to S3/GCS
  • IoT/sensors: MQTT → Kafka → lake
  • SaaS: Salesforce, HubSpot via Fivetran/Airbyte

🧠 Quick Check

A fintech company wants to detect fraudulent transactions within 2 seconds of them occurring. Should they use batch or streaming ingestion? Why?

Streaming ingestion — because fraud detection requires sub-second latency. Batch (e.g., hourly jobs) would process the fraud too late. Tools: Kafka + Flink or Kafka Streams. Latency target of <2s rules out any batch approach.

Chapter 2 · 25 min
Data Storage — Lakes, Warehouses & Lakehouses
▼
OLTP OLAP Data Lake Lakehouse Partitioning
🏗️ Storage Architecture Evolution

RDBMS (1980s–) OLTP

Row-oriented. Fast writes/point queries. Postgres, MySQL. Not built for analytics at scale.

Data Warehouse (1990s–) OLAP

Column-oriented. Optimised for aggregations. Redshift, Teradata. Structured data only.

Data Lake (2010s–) Raw

Schema-on-read. Stores raw files (CSV, JSON, Parquet) on cheap object storage (S3). Risk: swamp.

Lakehouse (2020s) Best of Both

Delta Lake, Iceberg, Hudi. ACID on data lakes. Schema enforcement + cheap storage.

📐 Storage Layer Comparison
FeatureData WarehouseData LakeLakehouse
SchemaSchema-on-writeSchema-on-readBoth
ACID✅ Yes❌ No✅ Yes
Data typesStructuredAnyAny
CostHighLowLow-Med
ExampleSnowflake, BigQueryS3 + HiveDatabricks, Delta

File Formats Matter

  • CSV/JSON — human-readable, large, no compression by default
  • Parquet — columnar, compressed, ideal for analytics queries
  • ORC — similar to Parquet, preferred in Hive/Presto
  • Avro — row-based, great for schema evolution in streaming

Partitioning Strategy

Partitioning splits data into folders by a key (e.g., date=2024-01-15/country=IN/). Queries that filter on the partition key skip irrelevant files entirely — called partition pruning.

🧠 Quick Check

Your ML team needs to store raw image files, JSON event logs, and processed feature tables. Which storage architecture fits best?

Lakehouse (e.g., Delta Lake on S3). Raw images and JSON go into the lake zone. Processed features are written as Delta tables with schema enforcement. Supports both unstructured + structured data with ACID guarantees.

Data Modeling — Structuring for Usability
▼

A data model is the blueprint that determines how tables relate. Good models make queries fast, intuitive, and maintainable.

⭐ Star Schema (Most Common in Warehouses)
📅 Dim: Date
👤 Dim: Customer
→
⭐ FACT: Sales
revenue, qty, discount
←
📦 Dim: Product
🏪 Dim: Store

Fact table = measurements. Dimension tables = context for slicing.

Modeling Approaches

  • Kimball (Star Schema): Denormalized. Fast queries. Great for BI.
  • Inmon (3NF): Normalised warehouse. Slower queries but flexible.
  • Data Vault: Hubs, Links, Satellites. Audit-friendly, handles change well.
  • OBT (One Big Table): Fully flat. Fastest queries, highest storage cost.

Slowly Changing Dimensions (SCD)

What if a customer changes their city? SCD Type 2 keeps history by adding a new row with validity dates. Type 1 overwrites. Type 3 stores old+new in columns.

dbt — The Modern Transform Layer

dbt (data build tool) lets analysts write transformations in plain SQL. It handles dependency resolution, testing, docs, and versioning. A typical dbt project has:

  • models/staging/ — clean raw tables
  • models/intermediate/ — business logic
  • models/marts/ — final analytical tables

Orchestration with Airflow

Apache Airflow schedules and monitors pipelines as DAGs (Directed Acyclic Graphs). Each node in the DAG is a task (SQL query, Python script, Spark job). Failures trigger alerts and retries automatically.

🧠 Quick Check

A startup uses BigQuery and their team only knows SQL. Should they use ETL or ELT? What tool for transformations?

ELT — load raw data into BigQuery first, then transform using dbt. BigQuery's compute is cheap at scale, and dbt lets SQL-fluent analysts own transformations without needing Python/Spark.

Chapter 5 · 20 min
Metadata Management & Data Cataloging
▼

Without a catalog, engineers spend 30%+ of their time just finding data. Metadata management answers: what exists, where is it, who owns it, what does it mean?

Types of Metadata

Technical

Schema, data types, row count, null %, column stats, partitions

Business

Descriptions, glossary terms, business owner, data domain, SLA

Operational

Last refreshed, job history, query frequency, cost per query

Lineage

Where did this column come from? What depends on it?

Popular Catalog Tools

  • DataHub (LinkedIn): Open source, rich lineage, API-first
  • Apache Atlas: Hadoop ecosystem, governance-first
  • Alation: Enterprise, AI-powered search
  • dbt Docs: Auto-generated from your dbt project

Testing with Great Expectations

Great Expectations lets you define expectations like assertions on your data. These run in your pipeline and halt it if violated.

# Example expectation
expect_column_values_to_not_be_null("user_id")
expect_column_values_to_be_between("age", 0, 120)
expect_column_values_to_be_unique("order_id")
expect_column_pair_values_to_be_equal("price", "total/qty")

Anomaly Detection

Beyond static checks, tools like Monte Carlo and Soda use ML to detect distribution shifts — when row counts drop 40%, when a column's mean drifts, or when fresh data stops arriving.

🧠 Quick Check

You find that 15% of rows in your orders table have NULL customer_id. Which quality dimension is violated and what should you check?

Completeness is violated. Investigate: upstream ingestion bug? CDC missed events? Source system allowing NULLs? Add a not_null expectation in dbt tests or Great Expectations to catch this going forward.

Data Lineage & Observability
▼

Lineage tracks the provenance of data — every transformation it went through. Critical for debugging broken dashboards and for compliance.

🔗 Column-Level Lineage Example

→
revenue_report
name column
If CRM schema changes → trace impact instantly through all downstream tables

Observability vs Monitoring

  • Monitoring: Known metrics, thresholds (row count > 0)
  • Observability: Unknown unknowns — ML detects anomalies you didn't anticipate

The 5 Pillars of Data Observability

  • Freshness: When was the table last updated?
  • Volume: Expected row count in range?
  • Schema: Did column types change?
  • Distribution: Value ranges shifted?
  • Lineage: Upstream tables healthy?

Chapter 8 · 12 min
Data Security & Access Control
▼

RBAC

Role-Based Access Control. Grant roles (analyst, engineer) not individuals.

Column Masking

Show SSN as ***-**-1234 to unauthorised users. Supported in Snowflake, BigQuery.

Row-Level Security

User X can only see rows where region = 'India'. Enforced by the warehouse.

Encryption

At-rest (AES-256) and in-transit (TLS). Cloud providers handle most of this.

Audit Logging

Record who queried what, when. Required for SOC2, HIPAA, GDPR compliance.

Tokenisation

Replace sensitive values with tokens. Original value stored in a vault, not the warehouse.

Data Governance — Policies & Stewardship
▼

Governance is the system of accountability over data assets. It's not a tool — it's people, processes, and policies.

Core Elements

🏛️ Data Governance Framework
Data Council — executive oversight, priority setting
Data Owners — domain leaders responsible for datasets
Data Stewards — day-to-day quality and definition management
Data Engineers — technical implementation of policies
Data Consumers — analysts, scientists, product managers

Compliance Frameworks

  • GDPR: EU regulation — right to erasure, consent management
  • CCPA: California — consumer data rights
  • HIPAA: US healthcare — PHI protection
  • SOC 2: Trust criteria for SaaS — audit trails, access logs

Data Classification

Classify data by sensitivity: Public → Internal → Confidential → Restricted. Access controls, retention, and encryption tighten at each level.

Semantic Layer

A semantic layer sits between the warehouse and BI tools. It defines business metrics once — "revenue = sum(amount) where status='paid'" — so every tool shows the same number.

Feature Stores

For ML teams, a feature store (Feast, Tecton) serves precomputed features to models. It solves the training-serving skew problem — the same feature logic runs in batch (training) and real-time (inference).

Reverse ETL

Moving data back from the warehouse into operational tools. Example: sync a "churn risk score" from BigQuery → Salesforce so the sales team acts on it without opening another tool. Tools: Census, Hightouch.

Lifecycle Policies

  • Retention: How long to keep data (GDPR requires deletion of PII after purpose)
  • Tiering: Auto-move data from hot → warm → cold based on access patterns
  • Deletion / Right to Erasure: Ability to delete a user's data across all tables
  • Versioning: Time-travel in Delta Lake / Iceberg — query data as of any past point

Cloud Storage Tiers (AWS)

  • S3 Standard — Hot. Immediate access. ~$23/TB/month
  • S3 Infrequent Access — Warm. ~$12.5/TB/month
  • S3 Glacier Instant — Cold archive, millisecond retrieval. ~$4/TB/month
  • S3 Glacier Deep Archive — Coldest. 12-hour retrieval. ~$1/TB/month

🧠 Final Review

A GDPR request comes in: "Delete all data about user ID 12345." Walk through which data management components are involved.

1. Catalog — locate all tables containing user 12345 data. 2. Lineage — find all derived tables downstream. 3. Governance — validate the deletion request meets policy. 4. Security / Access — confirm requester is authorised. 5. Lifecycle — trigger deletion across hot, warm, cold tiers. 6. Quality — verify post-deletion that no orphaned records remain. All 11 components touched in one request!
